# 架构初步设计

## 1. 系统划分

系统分为两个端：

- 用户端：登录、浏览影片、筛选场次、在线选座、提交订单、支付。
- 管理员后台端：登录、查看购买情况、查看订单、调整场次票价。

## 2. 核心场景

场景一：用户购票。

- 用户登录后选择影片和场次。
- 系统展示座位状态。
- 用户选择座位后提交订单。
- Redis 临时锁定座位，防止其他用户购买同一座位。
- 用户支付后，订单状态写入持久化数据，座位变为已售。

场景二：管理员运营。

- 管理员登录后台。
- 查看订单数、收入、售出座位。
- 查看最近订单。
- 调整场次票价，调价结果写入持久化数据。

## 3. 架构设计

表现层：

- Vue 单页应用。
- 用户端和管理员后台共用前端，但根据登录角色显示不同区域。
- ECharts 展示销售和收入图表。

接口层：

- Express REST API。
- 登录接口签发 token。
- 订单接口要求用户 token。
- 后台接口要求管理员 token。

业务层：

- 影片/场次服务。
- 座位锁服务。
- 订单服务。
- 支付确认服务。
- 后台运营服务。
- 本地持久化数据仓库。

基础设施层：

- `data/cinema-db.json`：用户、影片、场次、订单、已售座位、票价等真实业务数据。
- Redis：座位锁和事件队列。
- Prometheus：指标采集。
- Nginx、Docker、Kubernetes：部署和扩展配置。
- Spring 后端骨架：Spring Security、Shiro、读写分离、RabbitMQ、Sentinel、Nacos、Slf4j/logback、JUnit。
- Flask 推荐服务：独立推荐模块。

## 4. 可用性场景

热门座位并发购买：

- 多个用户同时购买同一座位。
- Redis 原子锁保证只有一个订单能锁定成功。
- 其他用户收到座位被锁定提示。

用户重复点击支付：

- 订单状态只能从 `PENDING_PAYMENT` 变为 `PAID` 一次。
- 已支付订单再次支付会被拒绝。

后台误操作控制：

- 调价接口仅管理员可访问。
- 普通用户访问后台接口返回权限不足。

Redis 故障：

- 系统降级为内存锁，保证单机演示不中断。
- 健康检查显示当前 Redis 模式。

## 5. 答辩说明

页面现在不展示多余架构说明，只保留真实业务功能。架构技术、部署配置和课程技术映射放在报告和工程配置文件中说明。

业务数据不是一次性模拟对象，而是写入 `data/cinema-db.json` 的持久化数据。用户购票、支付和管理员调价都会更新该文件，服务重启后仍可读取。
